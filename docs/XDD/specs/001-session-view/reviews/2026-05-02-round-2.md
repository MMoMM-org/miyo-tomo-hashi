### M1 — settings snapshot in run()
`const { vault, settings } = this` on line 156 destructures the live-getter into a stale local. M4-fix introduced the getter so settings changes mid-run are visible, but the destructure defeats it — line 465 still uses `this.settings` while lines 259/388/391 use the snapshot. **Fix:** drop `settings` from the destructure; use `this.settings.x` at every read site, or document an explicit single-snapshot policy.

### M2 — no-instances treated as transient
`startAutoReconnect` short-circuits only on `socket-permission-denied` and `daemon-unreachable`. `no-instances` falls through → 5 attempts, ~15 s of backoff, then `exhausted` with no useful message. **Fix:** add `err.code === "no-instances"` to the non-transient short-circuit (containers don't appear by retrying).

### M3 — peer checkbox ticked before applied flag
H5's batching moved `markActionsApplied` writes to post-loop. `tickPeerCheckbox` still fires per-action inside the loop. Plugin crash between tick and post-loop flush → peer `.md` shows `[x] Applied` while source JSON still has `applied: false`. Re-run will re-enqueue the action; user sees a ticked-but-still-pending state. **Fix:** move `tickPeerCheckbox` to the post-loop flush, or document that peer-sync is best-effort/ahead-of-truth.

### M4 — dispose race
`handleSessionClose` is called as `void this.handleSessionClose(...)` from `onClose`. Race window: `dispose()` runs `bumpEpoch()` first, then `this.disposed = true`, then awaits `teardownSession`. A remote-close firing in the window between `bumpEpoch` and `disposed=true` enters `handleSessionClose`, finds `disposed===false`, proceeds to `startAutoReconnect`. Loop will eventually observe the bumped epoch and exit, but state machine is left in `reconnecting` after `dispose()` returns. **Fix:** move `this.disposed = true` to be the first line of `dispose()`, before `bumpEpoch`.
