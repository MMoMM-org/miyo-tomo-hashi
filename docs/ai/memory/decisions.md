- **When a plan task and an SDD code sketch conflict, the plan wins (assuming the plan task explicitly references a newer ADR revision).**
  XDD leaves SDDs and plans as parallel artifacts that can drift after an ADR revision. Plans are usually rewritten in full at the end of `xdd-plan`; SDD code sketches only update when someone explicitly edits them. Spec 001 plan-task T1.3 said "drop `Readable<T>` and `derived<T,U>` per ADR-4 v3" while SDD `solution.md` lines 432-454 still showed `interface Readable<T>` + `class Store<T> implements Readable<T>`. Implementer flagged it; we followed the plan and patched the SDD in commit `4d039f2`.
  - **Why:** plan tasks for a phase are written *after* the latest ADR revisions are folded in (xdd-plan reads SDD + decisions log). SDD code sketches are illustrative; they decay when ADRs revise without an explicit sketch edit. The plan is the operational contract for the implementer.
  - **How to apply:**
    1. When an implementer agent flags a Plan ↔ SDD conflict, follow the plan and log the SDD as drift.
    2. Before each phase starts, scan for similar drift between the phase's task texts and the SDD sections they reference (grep for "ADR-N v[2-9]" or "simplification" in plan + diff against SDD code blocks).
    3. After implementation, patch the SDD in a follow-up commit so future readers don't repeat the resolution.
  - **Out of scope:** if an SDD revision is *newer* than the plan task (rare — usually plan is regenerated after SDD edits), the SDD wins and the plan needs an update. The default direction is plan → wins because plan is the more recently regenerated of the two in normal XDD flow.

- **Use an `epoch` counter to protect async state machines from races where external transitions can fire during in-flight async operations.**
  When a service exposes async methods like `connect()`, `disconnect()`, `forceReconnect()`, the user (or the runtime) can fire one while another is mid-flight. Without a guard, a slow `attach()` from a stale `connect()` can resolve AFTER a `disconnect()` and overwrite the current state — leaking a session and putting the store in the wrong state. `TomoConnection` (commit `ba065e2`) handles this by:
  1. Maintaining `private epoch = 0` on the service.
  2. Bumping it on every externally-initiated transition (`connect`, `disconnect`, `forceReconnect`, `autoReconnectIfRemembered`, `dispose`).
  3. Each in-flight async operation captures the current epoch at start. When it resolves, it compares the captured epoch against the live one. If different → the operation is stale; silently `close()` the freshly-acquired session and discard the result.
  - **Why:** simpler than AbortController everywhere; doesn't require the underlying API (dockerode `attach`) to support cancellation; keeps the state-store consistent without having to reason about every interleaving.
  - **How to apply:** any service with `connect/disconnect/reconnect`-style methods AND an underlying async API that returns a stateful resource (stream, socket, file handle) — bump epoch on transition; capture-and-compare on async resolve; clean up the resource if epoch moved. Document the discipline in the class header so future maintainers don't accidentally break it by skipping a bump.

- **GitHub issue organization: milestones = release versions, Track lives in the Project-board field, epics use native sub-issues + `needs-tomo`/Blocked-by.** Three axes settled after two false starts (thematic milestones, since reverted; and confusing the board Track field with the `area:` labels):
  1. **Milestones are versions, not themes.** `v0.1.x` = post-v0.1 hardening that needs no Tomo change (#45/#47/#24/#28/#46); `v0.2` = Tomo-contract apply-support (#37/#41–44, all `needs-tomo`-gated). Thematic milestones were tried and deleted — they mirror the `area:` labels 1:1 and, being open-ended, never reach a "done" close-out.
  2. **Track is a Project-board single-select field, not the `area:` label.** Board #3 "MiYo · Tomo ⇄ Hashi" carries cross-repo **Track** (MVP-Polish/Obsidian-Power/Post-Launch/Tech-Debt/Hashi-v0.1/Contract), **Area**, **Priority** (P0/P1/P2), **Status**. The `area:` issue-labels (executor/hooks/ide-bridge/build/session-view) are a separate Hashi-local axis. The two overlap but are not the same taxonomy, and the board fields do **not** auto-mirror the labels.
  3. **Epics = native GitHub sub-issues (membership) + `needs-tomo`/Blocked-by (dependency).** #37 is the reference: #41–44 are linked as sub-issues (which gives the progress bar) and each is blocked on its upstream Tomo contract via the `needs-tomo` label + `↑ Tomo #N` body refs. Cross-repo blockers cannot be native Blocked-by (GitHub issue dependencies are same-repo only), so the label carries them.
  - **Why:** both false starts came from conflating the board's strategic cross-repo Track with the Hashi-local `area:` labels, and from treating milestones as categories rather than releases. Versions give milestones a close-out moment; the board Track is the shared Tomo↔Hashi planning axis.
  - **How to apply:** a new issue needs **both** layers set — `area:` + `Pn-` labels AND the board Track/Area/Priority fields (found empty on #41–44 and entirely off-board for #45–47). Reading/writing board fields requires gh with `-s project` scope (`gh auth refresh -s project`). Put release intent in the milestone, never a theme; mirror Priority between label (`Pn-`) and board (P0/P1/P2) so they don't drift.

- **Executor QA test material must derive from real Tomo output, not synthetic fixtures hand-built on the Hashi side.** Fixtures invented on the Hashi side satisfy Hashi-internal coverage but can never surface Tomo↔Hashi drift — which is the actual integration risk. Two options: (a) generate material from scratch (Hashi-blind to drift) or (b) tell Tomo what instruction shapes we need and test against its real emission. Prefer (b); keep it simple.
  - **Why:** the value of executor testing is catching contract drift between what Tomo emits and what Hashi applies. Self-authored fixtures encode Hashi's *assumption* of the contract, so they stay green even when the real Tomo output would fail.
  - **How to apply:** drive executor scenarios from captured real `instructions.json` produced by a Tomo walk, or from a shape explicitly agreed with Tomo — not from fixtures written to make the executor pass. Close spec gates honestly with accepted-gap notes rather than fabricated green.

- **The run log is write-once: `RunLogWriter.start()` reserves the path but writes no file; `finalize()` creates it exactly once via `vault.create()`. Never reintroduce a start-time placeholder.** (#58 → 0.8.1)
  - **Why:** a placeholder written at `start()` IS the 190-byte "ghost log" an aborted run leaves behind, and the two-write lifecycle (create placeholder + `finalize` overwrite) doubles the modify events — letting external frontmatter automators win a write race and clobber a finalized log back to the placeholder (see `troubleshooting.md`). Write-once means an aborted run leaves no file, and a single `create()` halves the automator race surface. This superseded the #51 crash-safe try/catch, which couldn't help because no Hashi exception was thrown.
  - **Tradeoff (accepted):** a hard process-kill mid-run now leaves **no** log for that run (vs. a placeholder hinting "a run started at HH:MM"). In-Hashi exceptions are still covered — the abort `catch` writes a real error-row log via `finalize()`. The marginal "a run started" signal wasn't worth the recurring ghost-log confusion.
  - **How to apply:** keep ALL run-log file I/O inside `finalize()`. If you need a "run in progress" indicator, drive it from the state store / UI, never a file on disk. Related: `add_relationship`'s `skipped-already` now also writes `applied:true` (graduates) so already-correct notes stop reappearing every run.
